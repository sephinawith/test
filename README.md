二、 实践流程
 1. 环境配置： 在本地安装 Git 环境，并配置了全局的用户名和邮箱。
 2. 本地仓库创建： 在 VS Code 中打开“第三次实验”文件夹，初始化本地 Git 仓库，并使用 VS Code 的源代码管理工具对文件进行追踪。
 3. 远程仓库链接： 在 GitHub 上创建了公开仓库，通过 git remote add origin 命令将本地仓库与远程仓库建立连接。
 4. 代码推送： 将本地暂存区的文件进行 Commit 提交后，通过配置网络代理，使用 git push 命令成功将代码推送到 GitHub 云端。

四、 遇到的问题及解决方法
问题 1：执行 git push 时报错 fatal: No configured push destination.
原因分析： 本地仓库不知道要把代码推送到哪个具体的远程服务器地址。
解决方法： 在 GitHub 仓库页面复制了 HTTPS 链接，然后在终端执行 git remote add origin <链接>，成功为本地仓库指明了推送目标。
问题 2：VS Code 插件一直显示“正在加载...”，且终端推送时网络一直超时。
原因分析： 电脑虽然开启了 Clash 网络代理，但终端里的 Git 命令默认不会自动读取系统代理，导致直连 GitHub 时网络被阻断。
解决方法： 查阅资料后，在终端手动输入命令 git config --global http.proxy [http://127.0.0.1:7890](http://127.0.0.1:7890) 和 https.proxy 配置，强制 Git 走本地 7890 代理端口，顺利解决网络超时问题。
问题 3：远程仓库已有初始 README，导致本地推送被拒绝。
原因分析： 在 GitHub 创建仓库时勾选了“初始化 README”，导致云端和本地历史不一致，Git 为防止误覆盖拦截了推送。
解决方法： 确认云端文件无重要内容后，使用 git push -u origin main -f 命令进行了强制推送，用本地代码覆盖了云端仓库。
五、 Git 学习心得
经过这次完整的实践，我深刻体会到了版本控制对于计算机专业学生的重要性。以前写代码只能靠不停地“另存为”来备份，现在有了 Git，不仅每一次的修改都有迹可循，即使写出了 bug 也能轻松回退到之前的安全版本。虽然一开始面对终端命令行和各种报错（比如网络代理配置、远端冲突）觉得有些棘手，但把这些坑一个个填平后，对代码工程化的理解更深了一步。这为以后的多人协作开发和大型项目管理打下了很好的基础。
